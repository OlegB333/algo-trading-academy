# Бэктест стратегии в Freqtrade

## Важные правила

1. **Никогда не изменяй стратегию на основе результатов бэктеста без явного запроса ученика.**
   Если бэктест дал 0 сделок или плохой результат — объясни почему и спроси, что ученик хочет сделать.

2. **После бэктеста обязательно:**
   - Покажи ученику ключевые метрики в виде таблицы (Profit, Drawdown, Win Rate, кол-во сделок, Market change)
   - Объясни результаты простым языком
   - Направь в FreqUI для визуального анализа: *"Откройте http://localhost:PORT → раздел Backtest, чтобы увидеть график доходности и список сделок"* (подставь реальный порт из `.project-context`)

Ученик — архитектор. Он принимает решения, ты — исполняешь.

## Базовая команда

```bash
docker compose run --rm freqtrade backtesting \
  --config /freqtrade/user_data/config/config.json \
  --strategy ИмяСтратегии \
  --timerange YYYYMMDD-YYYYMMDD
```

## Параметры

| Параметр | Описание | Пример |
|----------|----------|--------|
| `--strategy` | Имя класса стратегии | `SampleStrategy` |
| `--timerange` | Период тестирования | `20260101-20260401` |
| `--stake-amount` | Размер позиции | `100`, `unlimited` |
| `--max-open-trades` | Макс. одновременных сделок | `3` |
| `--strategy-list` | Сравнить несколько стратегий | `Strat1 Strat2` |
| `--enable-protections` | Включить протекции | — |
| `--timeframe` | Переопределить таймфрейм стратегии | `1h`, `5m` |

⚠️ `--timeframe` перезаписывает значение из стратегии. Используй только если
данные за этот таймфрейм уже скачаны — иначе бэктест упадёт.
Безопаснее менять `timeframe` в самом файле стратегии.

## Примеры

**Простой бэктест (стартовый набор данных):**
```bash
docker compose run --rm freqtrade backtesting \
  --config /freqtrade/user_data/config/config.json \
  --strategy SampleStrategy \
  --timerange 20260105-20260405
```

**Сравнение двух стратегий:**
```bash
docker compose run --rm freqtrade backtesting \
  --config /freqtrade/user_data/config/config.json \
  --strategy-list SampleStrategy MyNewStrategy \
  --timerange 20260105-20260405
```

## Бэктест Фьючерсов на Крипто

Для фьючерсов на Binance всегда явно передавай выделенный конфигурационный файл:
```bash
docker compose run --rm freqtrade backtesting \
  --config /freqtrade/user_data/config/config_crypto_futures.json \
  --strategy ИмяСтратегии \
  --timerange YYYYMMDD-YYYYMMDD
```

## Как объяснять результаты ученику

После выполнения бэктеста ты **обязан** предоставить ученику структурированный отчет:

1. **Основные метрики:** Выведи небольшую таблицу с результатами (Total Profit %, Market Change, Win Rate, Max Drawdown, количество сделок).
2. **Аналитика (Интерпретация):** 
   - **Total Profit % vs Market Change:** Честно скажи, обыграла ли стратегия рынок. Стратегия имеет смысл только если ее профит выше `Market change`.
   - **Win Rate:** Объясни, что низкий Win Rate — это нормально при хорошем Risk/Reward (Profit Factor).
   - **Max Drawdown:** Обрати внимание на просадку. Сколько бы потерял ученик на самом дне?
3. **Призыв к визуализации в FreqUI:** Обязательно предложи ученику открыть **веб-интерфейс FreqUI (сообщи ему локальный адрес, например http://localhost:8080/ во вкладке Backtest)** для наглядного изучения графика доходности и точек входов/выходов на графике.
4. **Дальнейшие шаги:** Спроси ученика, какие параметры стратегии он бы хотел поменять или предлагает ли он запустить Hyperopt. Никогда не принимай решение об изменении правил входа/выхода самостоятельно!

## Предварительные условия (ОБЯЗАТЕЛЬНЫЙ АЛГОРИТМ)

**Перед КАЖДЫМ запуском бэктеста** выполни следующие шаги:

**Шаг 1.** Прочитай файл стратегии и определи:
- `timeframe` (например `1h`)
- Список пар из `--pairs` (если указаны явно) или из `pair_whitelist` в конфиге

**Шаг 2.** Определи источник данных по конфигу:
- `config.json` → Спот Binance, папка `data/binance/`
- `config_crypto_futures.json` → Фьючерсы Binance, папка `data/binance/`
- `config_forex.json`, `config_stocks.json`, `config_futures.json` (в корне `user_data/`) → Interactive Brokers, папка `data/interactivebrokers/`

**Шаг 3.** Проверь наличие файлов данных для каждой пары:
- Файлы лежат в `user_data/data/<exchange>/` в формате `ПАРА-ТАЙМФРЕЙМ.parquet` (или `.feather`)
- Пример: `user_data/data/binance/BTC_USDT-1h.parquet`

**Шаг 4.** Для каждой пары из стратегии:
- **Если данных НЕТ** → скачай автоматически (используй навык `data`), без вопросов к ученику
- **Если данные ЕСТЬ, но период < запрошенного `--timerange`** → досложи данные (расширь диапазон), запустив `download-data` за нужный период
- **Если скачать НЕ УДАЛОСЬ** (IB не запущен, лимит глубины, ошибка соединения) → предупреди ученика и объясни причину

**Шаг 5.** Только после успешной загрузки всех данных — запускай бэктест.

> **НЕ СПРАШИВАЙ** ученика "Скачать данные?". Просто скачай их и запусти бэктест. Ученик ожидает результат, а не диалог о подготовке.

## Hyperopt (оптимизация параметров)

Это продвинутый этап — после того как базовый бэктест работает.

Если стратегия содержит `IntParameter`/`DecimalParameter` с `optimize=True`:

```bash
docker compose run --rm freqtrade hyperopt \
  --config /freqtrade/user_data/config/config.json \
  --strategy ИмяСтратегии \
  --hyperopt-loss SharpeHyperOptLoss \
  --spaces buy sell \
  --epochs 100 \
  --timerange 20260105-20260405
```

Доступные loss-функции:
- `SharpeHyperOptLoss` — максимизирует Sharpe (рекомендуется для начала)
- `SortinoHyperOptLoss` — учитывает только downside volatility
- `OnlyProfitHyperOptLoss` — только прибыль
- `MaxDrawDownHyperOptLoss` — минимизирует просадку

## Частые ошибки

1. **Нет данных** → сначала `download-data` (skill `data`)
2. **Timerange за пределами данных** → уменьши диапазон или скачай больше данных
3. **Стратегия не найдена** → проверь имя класса внутри .py файла
4. **Мало сделок** → предложи ученику увеличить период тестирования (timerange) или спроси, хочет ли он смягчить условия входа. **Никогда не меняй сам код стратегии без его явной команды!**
